# 约束与禁止事项

## 否定约束

**原因**: 这些约束违反后会导致代码质量下降、安全风险或系统不稳定。

- **不要**使用 `any` 类型的 Java 等价物 (如裸 Object)
  
  **原因**: 失去类型安全，编译器无法检查类型错误，运行时容易出错。

- **不要**在 Controller 层写业务逻辑
  
  **原因**: 违反分层架构，业务逻辑分散难以复用和测试。

- **不要**直接修改 `target/` 目录下的文件
  
  **原因**: `target/` 目录是 Maven 编译输出目录，修改会被覆盖。

- **不要**提交 `.env` 或包含密码的配置文件
  
  **原因**: 敏感信息泄露可能导致生产环境被攻击。

- **不要**使用过时的 API (参考废弃标记)
  
  **原因**: 过时 API 可能存在安全漏洞，且未来版本会移除。

## 强制约束

### 数据库

**原因**: 规范 SQL 编写方式，便于维护和安全审核。

- 复杂 SQL 必须使用 XML 方式
  
  **原因**: XML 方式便于版本管理、SQL 审核和复杂逻辑维护。

- 禁止在 Service 层使用 `LambdaUpdateWrapper.setSql()` 等原生 SQL 方法
  
  **原因**: 避免 SQL 拼接风险，应使用 Mapper XML 或 MyBatis Plus 安全方法。

### 架构

**原因**: 保证系统性能和可维护性。

- Controller 不直接访问数据库
  
  **原因**: 违反分层架构，增加耦合度，难以单元测试。

- 所有 IO 操作必须异步或使用连接池
  
  **原因**: 避免阻塞主线程，提高系统吞吐量和响应速度。
