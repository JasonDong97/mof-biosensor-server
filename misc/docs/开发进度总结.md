## 一、需求概述

### 1.1 项目背景

本项目旨在开发一款配合智能呼气检测硬件设备使用的小程序及配套云平台。用户通过小程序连接硬件设备，完成呼气检测，实时获取检测结果（浓度值、健康等级等）及健康建议。云平台负责用户数据存储、设备管理、数据分析，并为运营人员提供数据看板。

### 1.2 核心目标

- **核心功能**：实现小程序与硬件设备的稳定蓝牙通信，完成检测流程，准确解析并展示检测结果
- **用户体验**：提供简洁流畅的检测引导、清晰的结果展示、便捷的历史数据查询与趋势分析
- **数据闭环**：建立完善的用户 - 设备 - 检测数据管理体系，确保数据不丢失、可追溯、可分析
- **运营支持**：为运营团队提供多维度的数据统计看板，支持业务决策与异常监控
- **安全合规**：严格遵守数据安全与隐私法规，保障用户敏感信息安全

### 1.3 功能模块划分

根据产品需求文档（PRD），后端功能主要分为以下模块：

| 模块 | 章节 | 描述 | 优先级 |
|------|------|------|--------|
| **认证模块** | 3.3.2 | 登录、注册、Token 管理 | P0 |
| **用户信息管理** | 3.3.3 | 用户资料同步、更新 | P1 |
| **检测记录上报** | 3.3.4 | 单条/批量上报检测数据 | P0 |
| **检测记录拉取** | 3.3.5 | 分页查询、筛选历史记录 | P1 |
| **设备绑定同步** | 3.3.6 | 设备绑定、解绑、列表查询 | P0 |
| **反馈接口** | 3.3.7 | 用户反馈提交与查询 | P2 |
| **后端服务** | 3.4 | 用户服务、设备服务、数据服务、反馈服务、系统管理 | P1 |
| **运营数据看板** | 3.5 | 数据统计、可视化展示 | P2 |
| **数据安全与合规** | 3.6 | 加密、脱敏、权限控制 | P0 |

### 2.3 命名规范

#### 类命名
- **Controller**: `{功能}Controller` - 如 `AuthController`
- **Service 接口**: `I{功能}Service` - 如 `ISysUserService`
- **Service 实现**: `{功能}ServiceImpl` - 如 `SysUserServiceImpl`
- **Mapper 接口**: `{实体}Mapper` - 如 `SysUserMapper`
- **DTO**: `{功能}{DTO/AddDTO/UpdateDTO}` - 如 `WxLoginDTO`
- **VO**: `{功能}VO/{功能}ListVO` - 如 `UserInfoVO`

#### 方法命名
- **查询单个**: `get{名称}By{条件}` - 如 `getUserById`
- **查询列表**: `list{名称}` - 如 `listUsers`
- **分页查询**: `page{名称}` - 如 `pageRecords`
- **新增**: `add{名称}` / `save{名称}` - 如 `addUser`
- **更新**: `update{名称}` - 如 `updateUser`
- **删除**: `delete{名称}` - 如 `deleteUser`

#### 变量命名
- **成员变量**: camelCase - 如 `private String userName`
- **常量**: UPPER_SNAKE_CASE - 如 `TOKEN_TIMEOUT`
- **数据库字段**: snake_case - 如 `user_name`

### 2.4 代码规范要点

#### 导入顺序
1. 标准 Java 类库（`java.` 开头）
2. 第三方库（`javax.`、`cn.`、`com.` 等）
3. 项目内部类（`com.miqroera.biosensor.` 开头）

#### 注释规范
- **文件头注释**: 包含类名、描述、作者、时间
- **方法注释**: 包含参数说明、返回值说明
- **使用 OpenAPI3 注解**: `@Schema`、`@Operation` 等

#### 响应格式
```java
R.ok()                          // 无返回数据
R.ok(data)                      // 带数据返回
R.ok(data, "自定义消息")         // 带数据和消息
R.fail("错误消息")               // 失败响应
```

#### 事务规范
- 默认使用 `@Transactional(rollbackFor = Exception.class)`
- 了解不同传播行为：`REQUIRED`（默认）、`REQUIRES_NEW`、`NOT_SUPPORTED`

#### 日志规范
```java
log.info("项目信息：{}", projectId);  // 使用占位符
log.error("错误信息：{}", errorMsg, exception);
```

### 2.5 数据库设计

#### 表命名规则
- 前缀：`t_` 或 `sys_`
- 格式：snake_case
- 示例：`t_device`, `t_record`, `sys_user`

#### 字段规范
- **主键**: `id` (bigint, 自增)
- **创建时间**: `create_time`
- **更新时间**: `update_time`
- **删除标志**: `del_flag` (char(1), '0'正常 '1'删除)
- **创建者/更新者**: `create_by`, `update_by`

```java
// Refresh Token 生成逻辑
private String generateRefreshToken(Long userId) {
    String refreshToken = "rt_" + userId + "_" + UUID.randomUUID().toString();
    String redisKey = REFRESH_TOKEN_PREFIX + refreshToken;
    redisUtil.set(redisKey, userId.toString(), REFRESH_TOKEN_TIMEOUT);
    return refreshToken;
}

// Refresh Token 验证逻辑
private Long verifyRefreshToken(String refreshToken) {
    String redisKey = REFRESH_TOKEN_PREFIX + refreshToken;
    Object value = redisUtil.get(redisKey);
    if (value == null) return null;
    return Long.parseLong(value.toString());
}
```

### 5.3 Token 双令牌机制

```
┌──────────────────────────────────────┐
│          用户登录成功                 │
└─────────────┬────────────────────────┘
              │
              ▼
    ┌─────────────────────┐
    │  生成 Access Token   │ 有效期：7 天
    │  生成 Refresh Token  │ 有效期：30 天
    └──────────┬──────────┘
               │
               ▼
    ┌─────────────────────┐
    │  Access Token       │ → 存储在客户端
    │  (Bearer Token)     │   用于接口鉴权
    └─────────────────────┘
    
    ┌─────────────────────┐
    │  Refresh Token      │ → 存储在 Redis
    │  (auth:refresh:{})  │   用于刷新 Access Token
    └─────────────────────┘
```

**刷新流程**:
1. Access Token 过期后，客户端使用 Refresh Token 调用刷新接口
2. 服务端验证 Refresh Token（从 Redis 查询）
3. 验证通过后，注销旧登录态，重新登录
4. 生成新的 Access Token 和 Refresh Token
5. 返回给客户端

## 八、开发注意事项

### 8.1 常见坑点

1. **Sa-Token 排除路径**
   - 问题：配置 context-path 时，排除路径需要加前缀
   - 解决：`/api/v1/auth/**` 而不是 `/auth/**`

2. **Refresh Token 验证**
   - 问题：Refresh Token 被误用为 Access Token
   - 解决：明确区分两种 Token 的用途，Refresh Token 仅用于刷新接口

3. **手机号脱敏**
   - 问题：数据库中存储明文，接口返回时脱敏
   - 解决：在 VO 层进行脱敏处理，不在 Entity 层处理

4. **事务失效**
   - 问题：同类方法调用导致 `@Transactional` 失效
   - 解决：使用 `AopContext.currentProxy()` 或重构代码结构

### 8.2 代码审查要点

- [ ] 所有接口是否有统一的响应格式 `R<T>`
- [ ] 是否有适当的日志记录（info/error 级别正确使用）
- [ ] 数据库操作是否有事务控制
- [ ] 参数校验是否完整（@Validated + @NotBlank 等）
- [ ] 敏感数据是否脱敏
- [ ] 是否有 SQL 注入风险
- [ ] 异常处理是否合理（不吞异常、不暴露堆栈）
